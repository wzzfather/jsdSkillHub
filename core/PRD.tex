\section{产品概述}

基于 MRD 分析，本产品为企业内部提供 AI Skill 的统一管理平台。当前已实现：用户认证（JWT + 邮箱验证）、应用市场（搜索 + 分类 + 排序）、Skill 上传与三层安全扫描、审批工作台、管理员应用管理（上下架）、一键安装到 OpenClaw。

\section{关键交互流程}

\subsection{Skill 上架流程}

上传 zip $\to$ scanning（三层扫描）$\to$ pending\_review（审批工作台）$\to$ published（上架）/ rejected（驳回）

\subsection{Skill 下架流程}

管理员下架（填原因）$\to$ offline（我的应用显示已下架）$\to$ 管理员重新上架 $\to$ pending\_review（重新审批）

\subsection{开发者重新提交}

rejected 状态 $\to$ 开发者点击「重新提交」$\to$ pending\_review（重新进入审批队列）

\subsection{用户安装流程}

浏览市场 $\to$ 查看详情 $\to$ 下载 zip / 安装到 OpenClaw

\section{设计规范}

\begin{itemize}[nosep]
  \item \textbf{颜色方案}：黑白灰极简风格，主色 \texttt{\#1a1a2e}
  \item \textbf{状态标签颜色}：scanning=warning，pending\_review=info，published=success，offline=danger，rejected=danger
  \item \textbf{UI 框架}：Element Plus
\end{itemize}

\section{约束与规则}

\begin{itemize}[nosep]
  \item \textbf{工具链锁定}：仅 OpenClaw + Cursor，Claude/Codex 暂不参与
  \item \textbf{拍板机制}：有歧义时必须由魏子政拍板，禁止自行假设
  \item \textbf{文档一致性}：PRD 与 MASTER 不一致时，局部细节以 MASTER 为主，重大流程需拍板
\end{itemize}

\section{文档一致性维护规则}

当 PRD 与 MASTER 总需求文档内容不一致时，按以下规则处理：
\begin{enumerate}[nosep]
  \item \textbf{接口文本、功能描述、技术细节等局部不一致}：以 MASTER 总需求文档为主，PRD 同步修正
  \item \textbf{涉及重大流程或业务流转路径的问题}：必须由魏子政拍板决定方向，PM 不得自行假设
  \item \textbf{MASTER 未覆盖但 PRD 已定义的内容}：保留 PRD 内容，同步补充到 MASTER
  \item 每次修正后，同步更新 MASTER 和 PRD 的版本号
\end{enumerate}
