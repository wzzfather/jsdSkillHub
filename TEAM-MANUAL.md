# 团队协作说明书

> **给魏子政的使用指南** — 如何给 PM 和 Cursor 下需求、看进度、反馈问题

---

## 一、角色分工

| 角色 | 是谁 | 做什么 | 你怎么跟他沟通 |
|------|------|--------|--------------|
| **产品经理 (PM)** | 小助手（OpenClaw） | 需求生成、需求拆解、独立验收 | 直接跟我说 |
| **开发工程师** | Cursor Agent | 前端（含 UI 设计）+ 后端 + 环境 + 运维 + 测试（自验收+响应PM验收） | 我来调度，你不需要直接操作 Cursor |

> **注意**：Cursor 的测试 ≠ PM 的验收。
> - Cursor 负责**自验收**（根据自己生成的代码自测）和**响应 PM 验收**（PM 提出的问题必须回应修复）
> - PM 负责**独立验收**（站在用户角度，对照需求清单检查，不依赖 Cursor 的自测结果）

**简单说：你只需要跟我对话，我来调度 Cursor。**

## 二、日常需求怎么说

### 场景 1：要加新功能

> 你说：「我要加一个 xxx 功能，具体要求是……」

我会：
1. 把需求记到 memory
2. 拆解成开发任务
3. 分配给 Cursor 执行
4. 审计结果后向你汇报

### 场景 2：页面不好看 / 要改 UI

> 你说：「这个页面不好看，参考 xxx 的风格改一下」

我会：
1. 基于 UI 设计规范出修改方案
2. 把方案交给 Cursor 实现
3. 完成后给你看效果截图

### 场景 3：发现 Bug

> 你说：「xxx 功能有问题，点击 xxx 之后报错了 / 没反应 / 数据不对」

我会：
1. 记录 Bug（模块、复现步骤、优先级）
2. 分配给 Cursor 修复
3. 修复后重新测试，确认通过后汇报

### 场景 4：查看进度

> 你说：「现在开发到哪了？」

我会：
1. 查看进度表
2. 汇报每个模块的完成状态
3. 标注阻塞项和风险

### 场景 5：要改需求 / 方向调整

> 你说：「之前定的 xxx 方案要改，改成……」

我会：
1. 更新需求文档
2. 评估对已完成工作的影响
3. 制定调整计划
4. 继续推进

## 三、反馈链路

```
你提需求
  ↓
PM（我）接收 → 记录 → 拆解 → 分配
  ↓
Cursor 执行开发
  ↓
PM 审计 → 测试 → 通过？
  ├→ 是 → 向你汇报「完成」
  └→ 否 → 退回 Cursor 修复 → 重新测试 → 直到通过
  ↓
你验收 → 满意？
  ├→ 是 → 标记完成
  └→ 否 → 反馈问题 → PM 记录 → 退回处理
```

**你只需要在两个节点参与**：
1. **提需求时**：说清楚要什么
2. **验收时**：看结果，给反馈

中间的过程我全包了。

## 四、需求描述的最佳方式

好的需求描述 = 我能直接转给 Cursor 执行。包含以下要素：

### 最简版（一句话也行）
> 「加一个应用列表页，能按分类筛选，能搜索」

### 完整版（复杂需求推荐）
> 「我要加一个审批功能：
> 1. 开发者提交应用后进入待审核状态
> 2. 审批人能看到待审核列表
> 3. 点击详情能看到应用信息 + 扫描报告
> 4. 能通过或驳回，驳回要填理由
> 5. 不同角色看到不同的按钮（开发者看不到通过/驳回）」

### 改版/反馈版
> 「这个页面有几个问题：
> 1. 按钮颜色不对，应该是绿色不是蓝色
> 2. 表格太挤了，间距要加大
> 3. 缺少一个导出功能
> 参考飞书应用市场的风格」

## 七、常见问题

**Q：我能直接操作 Cursor 吗？**
A：不需要。你跟我说需求，我来调度 Cursor。如果你想自己操作也可以，但需要先读 `CLAUDE.md` 和 `skills/developer/SKILL.md`。

**Q：如果我说的需求不清楚怎么办？**
A：我会追问确认，不会盲目执行。但描述越清晰，推进越快。

**Q：开发过程中能改需求吗？**
A：随时可以。我会评估影响范围，告诉你哪些已完成的工作需要调整。

**Q：怎么知道 Cursor 在干什么？**
A：我会实时汇报进度。你也可以随时问"现在到哪了"。

**Q：如果我对结果不满意？**
A：直接跟我说哪里不对，我会记录问题并安排修复。你不需要知道具体是 PM 还是 Cursor 的问题，你只管提反馈。
